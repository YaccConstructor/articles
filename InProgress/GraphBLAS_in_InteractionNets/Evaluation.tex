\subsection{Experimental Setup}

\paragraph{Hardware.}
All experiments were conducted on a system with an AMD Ryzen 9 7900X 12-core processor (24 threads with hyper-threading enabled, base clock 4.7 GHz) and 128 GiB of DDR5 RAM.

\paragraph{Software.}
We compare four implementations:
\begin{itemize}
    \item \textbf{F\#}: Our quadtree-based implementation, compiled with F\# !!! and the .NET !!! runtime. This serves as a baseline representing a conventional functional language approach. Single thread. Benchmarkdotnet for benchmarking automation.
    \item \textbf{Inpla}: Our interaction-net-based implementation, executed on the Inpla runtime system (version !!!). Settings: !!! Patched version of Inpla. Fixes in Inpla: list reading, !!!!
    \item \textbf{Vine}:  Our interaction-net-based implementation, executed on the Vine runtime system (version !!!). Settings: 110Gb heap per worker.
    \item \textbf{LAGraph}: A collection of graph algorithms built on SuiteSparse:GraphBLAS (version !!!), providing state-of-the-art, linear-algebra-based reference implementations of BFS, SSSP, and triangle counting.
    \item \textbf{NetworkX}: A popular Python library for graph analysis (version !!!), included as a baseline representing ease of use over optimization.
\end{itemize}

\paragraph{Research Questions.}
Our evaluation addresses the following research questions:

\begin{description}
    \item[RQ1 (Correctness):] Do the F\# and Inpla implementations produce correct results for BFS and triangle counting on all datasets, when compared against reference outputs from NetworkX and LAGraph?

    \item[RQ2 (Performance -- Inpla vs. Vine vs. State-of-the-Art):] How does the execution time of Inpla compare to LAGraph on BFS, SSSP, and triangle counting?
         We also include F\# and NetworkX as additional baselines representing different points in the performance--productivity trade-off.

    \item[RQ3 (Parallel Scaling):] How well do the Inpla, Vine, and LAGraph implementations scale with the number of available CPU cores?
          We measure speedup from 1 to all available cores for BFS, SSSP, and triangle counting, with hyper-threading both enabled and disabled.

    \item[RQ4 (Impact of Matrix Reordering):] Matrix reordering techniques (e.g., reverse Cuthill--McKee) can improve locality and cache performance.
    \item[] Even for ~\cite{10.1145/3731599.3767441}
          How does applying such reordering affect the performance of Inpla, F\#, and LAGraph implementations?
\end{description}
